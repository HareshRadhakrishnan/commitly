# ADR-0011: RAG Context Pipeline for AI-Generated Explanations

## Status

Accepted

## Context

The original `process-push` pipeline generated marketing content (changelog, LinkedIn, Twitter) using only commit messages. This produced generic output because the LLM had no understanding of:

1. **What actually changed** â€” diffs were fetched for significance checking but then discarded before content generation.
2. **What the repository does** â€” no domain context was stored per project; the LLM treated every repo identically.
3. **What surrounding code a change belongs to** â€” even when a diff was provided, the LLM saw only `+/-` lines without knowing what function, module, or system the change affects.

These gaps produced marketing copy that was structurally correct but semantically shallow â€” indistinguishable from output generated by reading commit messages alone.

## Decision

Implement a full RAG (Retrieval-Augmented Generation) context pipeline with four components:

### 1. Repo indexing on connect (`index-repo` Inngest function)

When a user connects a repo, fire `github/repo.connected` and run a background job that:
- Fetches the recursive file tree (up to 150 source files, filtered by extension and size)
- Splits each file into function/class-level chunks using heuristic boundary detection (`BOUNDARY_PATTERNS` in `chunker.ts`)
- Stores `start_line`, `end_line`, and `content_sha` per chunk alongside the embedding
- Embeds all chunks via OpenAI `text-embedding-3-small` and stores in a new `repo_file_chunks` pgvector table
- Summarizes `README.md` with GPT and stores as `projects.repo_summary`

### 2. Significance check moved first

`check-significance` is moved immediately after `fetch-diffs`, before any expensive retrieval steps. Pushes assessed as insignificant (typos, refactors, config changes â€” ~60-70% of all pushes) exit early, skipping embedding calls and GPT explanation entirely.

### 3. Function-level context retrieval with 4-level fallback (`getFileContext()`)

For each changed file in a significant push:
1. Parse `@@` hunk headers to extract changed line numbers
2. Query `repo_file_chunks` for chunks whose `start_line`/`end_line` overlaps â†’ returns exact changed function chunks (no GitHub API call needed)
3. If no chunks found (new file, unindexed) â†’ fetch full file from GitHub API (cap 2000 chars)
4. If fetch fails â†’ use diff text only
5. If diff is empty â†’ return empty string (caller uses commit message only)

Deleted files (`status === 'removed'`) skip all levels and call `handleDeletedFile()` to purge their stale chunks.

### 4. Incremental re-index on push (`update-embeddings` Inngest function)

After each push, fire `github/repo.files-changed` as a non-blocking side-effect. `update-embeddings`:
- Deletes chunks for removed files
- Re-chunks and re-embeds modified/added files, with SHA-based cache to skip unchanged content
- Re-summarizes `README.md` only if it was in the changed set

### 5. New `explain-commits` step

Between `retrieve-context` and `generate-content`, a batched GPT call receives:
- Changed function chunks (from the vector store)
- Top-3 semantically related chunks (cosine similarity search)
- `projects.repo_summary`

It returns a plain-English explanation per commit covering: what changed, why it matters, any risks or improvements. Explanations are stored as `commit_explanations` in `ai_content` JSONB and displayed on the review page.

## Consequences

**Positive:**
- Marketing content is grounded in actual code semantics, not just commit messages
- The "What changed â€” engineer's view" card on the review page gives users a human-readable technical summary alongside the marketing copy
- Early significance check cuts retrieval and GPT costs for the majority of pushes
- SHA-based chunk caching prevents redundant re-embedding when file content is unchanged
- 4-level fallback ensures graceful degradation â€” the pipeline never hard-fails due to missing context

**Negative / Trade-offs:**
- Initial repo indexing adds latency on repo connect (30â€“90 seconds for a typical repo) â€” run in background via Inngest, not blocking the UI
- `repo_file_chunks` table grows proportionally with connected repos; IVFFlat index requires a minimum number of rows to be effective
- The `text-embedding-3-small` default is 1536 dimensions; reducing to 512 (3x cheaper) is possible if the `@ai-sdk/openai` version supports the `dimensions` parameter
- Function-level boundary detection uses regex heuristics â€” it works well for TS/JS/Python/Go/Rust but may produce oversized chunks for some file types
- `queryRelatedChunks` uses a Supabase RPC (`match_repo_chunks`) that must be manually created if cosine-similarity search is needed; the current implementation returns empty on error rather than failing

## Alternatives Considered

| Alternative | Why not chosen |
|-------------|----------------|
| Full file content passed to every LLM call | Sends 90%+ irrelevant code; hits context limits for large files; much more expensive |
| Store diffs in JSONB and query by commit SHA | Diffs only show what changed, not surrounding context; can't retrieve semantically related code from other files |
| Use a dedicated vector DB (Pinecone, Weaviate) | Premature at MVP scale; pgvector via Supabase eliminates a separate service and is sufficient for thousands of chunks per project |
| Re-run full repo index on every push | Too slow and expensive; incremental file-level updates with SHA cache achieve the same freshness at a fraction of the cost |
| Skip the explain-commits step; use diffs directly in generate-content | The generation prompt already existed; adding explanations as an intermediate representation produces richer, more specific marketing copy without changing the generation prompt's structure |
